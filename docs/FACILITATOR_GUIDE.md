# Facilitator Guide

Notes for the Innovation Team members running each table. Read this before the session.

---

## Your Role

You are a guide, not a builder. Your job is to keep the group moving, ask better questions, and stop them disappearing down rabbit holes. You should not be the one typing - that's on the trainees.

If a group is struggling technically, help them write a better prompt to Claude rather than fixing it yourself.

---

## The Session Flow

| Time | Your focus |
|---|---|
| 0–15 min | Run the discovery questions from [DISCOVERY.md](DISCOVERY.md). Push for specificity. |
| 15–20 min | Guide them through [DESIGN_BRIEF.md](DESIGN_BRIEF.md). Watch the scope - cut if needed. Then get them to run `/kickoff`. |
| 20–60 min | Stay out of the way unless they're stuck. Watch the clock. |
| 55 min | Hard stop on new features. Polish only from here. |
| 60–70 min | Help them prepare the demo story - a real scenario, a named user, a narrative. |
| 70–75 min | Get them to run `/present` and open `presentation.html` in the browser. |
| 75–90 min | Presentations. Each group gets 5 minutes, strict. |

**If a group arrives with a clear idea already formed**, skip Discovery and Design Brief. Tell them: "You're in build mode. Describe your idea to Claude, then run `/kickoff`." Don't make them work through process they don't need — that's 15 minutes they could spend building.

---

## Keeping Groups on Track

If they're stuck on the problem for too long (more than 15 minutes), help them pick something. A well-executed simple idea beats a brilliant idea they never start building. Suggest a use case from [USE_CASES.md](USE_CASES.md) if they're completely blank.

If they're over-scoping, point to the MVP scope table in the design brief. Ask: "If you only built the first user story, would that be demoable?" Usually yes. Cut everything else to nice-to-have.

If the build is going badly, that's normal. Help them write a clearer prompt, or suggest reverting the last change and trying a different approach. A simple demo that works is better than an ambitious one that doesn't.

If they're ahead of schedule, encourage them to improve the UI, add nice-to-haves, or sharpen the demo story. Don't let them add entirely new features after the 55-minute mark.

If they're behind schedule, be honest about what can be cut. The presentation and the thinking matter more than the completeness of the build.

---

## The Presentation

Each group presents for 5 minutes. Enforce this - the energy in the room depends on it.

Encourage them to follow this structure:
1. The problem (30 seconds)
2. What they built (30 seconds)
3. Live demo (2 minutes)
4. What it would take to ship (1 minute)
5. The ask / next step (30 seconds)

The `presentation.html` file (generated by `/present`) will structure this for them.

---

## The End-of-Session Checklist

Walk through [SHIPPING_CHECKLIST.md](SHIPPING_CHECKLIST.md) with your group in the last few minutes. The right answers don't matter; the point is showing trainees that shipping real software inside a law firm involves a lot more than writing code. Spark a conversation, don't test them.

If a group's idea has real legs, pivot the conversation to [SHIPPING_EXAMPLE.md](SHIPPING_EXAMPLE.md) - a worked example of the detail (data classification tiers, the security review, the path to a pilot). Use it as the bridge from "we built a thing" to "here's what taking it further actually looks like inside a firm." Swap in your own firm's stack and process where it differs.

---

## Folder Setup Before the Session

The folder structure has two zones:

- **`docs/`** — facilitator reference material. Trainees shouldn't touch this.
- **Root** — the live workspace. `PLAN.md`, code files, and `presentation.html` all land here automatically.
- **`materials/`** — optional. If you're running a scenario-based session (sample contract, deal documents, email chains), drop those files here before trainees arrive. Tell them: "there are reference documents in materials/ — mention that to Claude when you kick off."

If you're not running a scenario, `materials/` doesn't need to exist. Groups who bring their own idea just work from the root.

---

## The Point of the Day

The trainees don't need to become developers. They need to understand that:
- They can identify real problems worth solving
- They can communicate those problems clearly enough for an AI to build something
- Getting something from idea to prototype is possible in hours, not months
- Taking it further requires a different set of conversations - with IT, risk, data, and a sponsor

That's the mental model shift. The code is just the proof of concept.
